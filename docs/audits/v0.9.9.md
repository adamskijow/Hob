# v0.9.9 grounded why/what-if audit

Date: 2026-07-23. Parent: v0.9.8.

## Customer outcome

Hob can now continue the planning conversation after showing a deterministic
day plan or weekly outlook. The owner can ask naturally why a task did not fit,
which assumptions were used, or what would need to change. The local model
chooses the relevant explanation, but Hob displays only facts and safe levers
that the deterministic engine saved with the result.

The owner can also test a temporary counterfactual such as “what if the audit
only took 30 minutes and I could work until 7?” Hob reruns feasibility and
labels the result as a what-if. It does not silently edit the task estimate,
work profile, adopted plan, or Calendar. A day what-if remains a proposal and
still requires explicit adoption.

## UX and onboarding contract

- Plans with deferred work and outlooks with risk teach the feature in context
  instead of requiring a command.
- `/help`, the feature tour, and the one-time release note provide examples for
  why, assumptions, safe changes, and what-if language.
- Explanations name the saved plan/outlook and generation time, end with
  “nothing changed,” and distinguish temporary testing from durable correction.
- A missing, expired, malformed, or wrong-target analysis asks for a fresh plan
  rather than guessing. Explanation context expires after 24 hours.
- “Another hour” is accepted only when the previous result had an explicit
  minutes budget. Otherwise Hob asks where that hour exists or for an absolute
  total, because inventing a Calendar opening would be misleading.

This improves planning onboarding without pretending to solve the still-open
fresh-install timezone, model-readiness, or durable-service-install journey.

## Architecture and safety boundary

1. A day plan or outlook stores one versioned explanation snapshot in ordinary
   local meta state: typed query inputs, compact result/item state, deterministic
   facts, bounded suggestions, and the generated result.
2. The model receives the owner’s question and those fact/suggestion ids. Its
   output is only a selection. Invented ids are discarded; a failed selection
   falls back to relevant stored facts.
3. A typed `what_if` may temporarily override available minutes, budget scope,
   energy, earliest/latest availability, outer work bounds, one known task’s
   duration, or split permission. The engine runs on cloned open tasks.
4. A day what-if persists only proposed sessions. Durable task metadata and
   profile settings remain unchanged. Adoption stays a separate explicit plan
   action.
5. While an explanation snapshot is active, a separate local-model pass audits
   mutation-shaped output as durable or counterfactual. It uses one explicit
   outcome field to avoid structured-output branch bias.
6. The guard distinguishes declarations/commands (“the audit takes 30 minutes
   now”, “buy milk tomorrow”, “drop the audit”) from counterfactual questions.
   If the guard is unavailable, malformed, or uncertain, the turn fails closed
   as a model outage and applies nothing.
7. Any real task or setting mutation invalidates the old analysis before a
   same-turn replan may save a fresh result.

The code contains no phrase matcher for the owner’s language. The model owns
the semantic distinction; deterministic code owns ids, bounds, time math,
facts, mutation authority, and explicit adoption.

## Verification evidence

- 431 deterministic Python tests cover typed parsing, reconciliation, target and
  bounds validation, mutation-free explanation, hallucinated-id rejection,
  guard failure, durable/counterfactual contrasts, displayed-list confinement,
  typed-date preservation, snapshot expiry, temporary proposal persistence, and
  unchanged durable task state.
- The real-model interpreter corpus expands from 102 to 109 cases, including
  why, safe-change negotiation, temporary task estimates and working bounds,
  durable schedule correction, durable drop, and a new capture after analysis.
- `evals.analysis_eval` exercises the full real-model service loop: constrained
  plan, grounded explanation, combined duration/window what-if, and durable
  state comparison.
- Full lock/syntax, Python, native Swift/EventKit/Xcode, exact-commit
  Ubuntu/macOS CI, backup, deployment, and live-health evidence are required
  before the release is considered shipped.

## Remaining risks and next increment

1. Turns near an active analysis may require an additional local generation.
   The 24-hour expiry bounds that latency cost; target-hardware soak evidence is
   still required for 1.0.
2. The declared 7B/14B support story remains inconsistent. v0.9.9 is gated on
   the 14B corpus until the 1.0 model-baseline decision is made.
3. Explanations cover the deterministic facts Hob actually has. They do not
   invent motivation, Calendar titles, estimate learning, or hidden preference.
4. The highest next product risk is permanent queue blockage and recovery.
   Timezone/DST and durable installation remain parallel P1 acceptance work.

## Decision

Release as v0.9.9 only after the exact release commit passes all gates, the live
database is backed up, the daemon is restarted onto that commit, queues and
schema are healthy, and one real Telegram why/what-if exchange confirms the
customer wording and no durable task or setting changed.
