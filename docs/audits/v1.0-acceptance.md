# Hob 1.0 acceptance audit

This is the release-candidate evidence matrix for Hob 1.0. It is intentionally
started before v0.9 releases so missing proof cannot be reconstructed from
memory at the end. A green unit suite is evidence for the behavior a test names,
not proof of the broader customer journey.

Current baseline: v0.8.0 is released and healthy; v0.9 is implemented in draft
PR #1 with 345 deterministic tests, a 72-case real-model evaluation, a signed
EventKit build, and green Ubuntu/macOS CI. The live database currently contains
zero plan runs, sessions, or start nudges, so the v0.8 execution loop is not yet
dogfooded and v0.9 is not releasable.

## Release decision

**Not a 1.0 release candidate.** The product has strong local correctness and
privacy boundaries, but evidence is missing for sustained execution-loop use,
fresh-install timezone correctness, a manual permanent-queue recovery drill,
DST edges, accessible end-to-end operation, supported-model quality, and a guided
durable install/upgrade journey. The LLM-native value also needs to progress
from natural intent classification to trustworthy explanation and negotiation
over deterministic plan/outlook results.

## Prioritized findings

### P0: the core execution and outlook loops lack sustained-use evidence

The live database has no plan runs or sessions. Tests cannot establish whether
proposal wording is legible, adoption feels safe, a start nudge is useful, or
plan-aware EOD asks the smallest honest question. v0.9 remains draft until the
v0.8 sequence is observed; 1.0 remains impossible until v0.9 itself receives a
week of real use.

### P1: first-run time correctness needs live proof

The audit found that `HOB_TIMEZONE` defaulted to UTC while chat setup asked
hours and days without confirming timezone. The stacked time-correctness
increment now derives an IANA zone from macOS, retains an explicit environment
override and UTC fallback, removes the unsafe New York plist placeholder, and
states the active timezone whenever setup starts. A clean install and system-
timezone change still require manual proof before this finding can close.

### P1: poison-delivery recovery needs a copied-data drill

Inbox and outbox processing is ordered and stops on the first failure, which is
correct for transient faults. A permanent parsing/store/Telegram failure is
now recoverable on the stacked queue branch through privacy-safe status and
explicit local retry or quarantine. Quarantine retains the row, never happens
automatically, and is reversible. Mutating commands require the daemon's
database lease, while success output distinguishes unapplied inbox state from
already-applied outbound state and warns that an outbound retry can duplicate a
remotely accepted message. Deterministic poison-row tests pass; the documented
copied-production-data and VoiceOver drill is still required before closure.

### P1: durable installation is implemented but not rehearsed

The stacked durable-install increment generates the exact secret-free
LaunchAgent and adds install, status, restart, and state-preserving uninstall
commands. Setup verifies Telegram credentials and model readiness before
installation. Update stops the daemon, writes a raw verified pre-migration
backup, and restores the prior loaded definition on failure. Calendar consent
and first-owner pairing remain explicit. Clean-install, update, rollback,
reboot/sleep, VoiceOver, and failure-injection evidence still gate closure.

### P1: DST policy is partially proven

The stacked increment classifies local wall times deterministically. Fixed tasks
in a spring gap or repeated fall hour are deferred with an explanation instead
of guessed; reminders label a skipped time or the first repeated occurrence;
digest/EOD once-per-date behavior is tested. Calendar periods already carry
explicit offsets, but recurrence, adopted-session, timezone-change, and manual
EventKit transition evidence remain before this finding can close.

### Resolved P1: the supported-model promise is evidence-backed

The complete exact corpus passes 75/75 on `qwen2.5:14b-instruct` and only 53/75
on the prior 7B default. The failures include completions, bulk actions,
deadlines, settings, planning, notes, negation, and typo correction being
misread as captures, so core patches would not make 7B a coherent safe tier.
The stacked supported-model increment makes 14B the only supported default,
requires explicit acknowledgement for an experimental override, and teaches
the official 9.0 GB artifact size, a conservative 24 GiB memory recommendation,
and the exact M5 Max 128 GiB release-eval target. Representative lower-memory
latency remains a manual RC measurement, not an unsupported quality promise.

### P1: grounded explanation needs model and live repair evidence

The stacked grounded-explanation increment records a bounded decision-time
artifact after each plan/outlook and adds a typed, read-only follow-up. Model
target hints are validated; causal facts and non-mutating options are rendered
deterministically; literal questions still work during a model outage; Calendar
identity and content are excluded. Deterministic state-equality, partial-split,
corruption, export, privacy, and adversarial-consent tests pass, as does the
75-case 14B model corpus. Fresh-owner, VoiceOver, CI, and five real disagreement-repair journeys remain before
this finding can close.

### P2: accessible fallback and release version drift were found early

This audit changed silent non-text Telegram updates into a clear text fallback
and lets media captions use the normal capture path. It also added a test tying
the package version to the runtime version. These fixes are prerequisites, not
proof of the full accessibility or upgrade journeys above.

### P2: system timezone and core DST ambiguity were found early

The time-correctness increment adds five deterministic regressions and raises
the stacked suite to 350 tests. It removes a misleading fresh-install default
and refuses to fabricate a fixed DST time, but is not release evidence until
the stacked PR passes its full model/native/CI gates and a clean macOS journey.

### P2: sustained storage and logs have no operating envelope

Inbox, outbox, action, digest, and plan history are retained indefinitely and
launchd logs are unmanaged by Hob. This may be acceptable for one owner, but
1.0 needs a measured multi-year-size fixture, latency thresholds, documented
retention, and a safe log-rotation recommendation before making that promise.

## Coherent 1.0 scope

1. Finish, release, and dogfood v0.9 without weakening its explicit consent or
   read-only Calendar boundaries.
2. Make first install time-correct and durable: timezone confirmation, complete
   launchd install/update/rollback validation, Telegram credential check, and a
   useful doctor result before the owner expects a digest.
3. Complete the manual and accessibility drills for the stacked deterministic
   queue recovery, privacy-safe status, explicit local retry/quarantine,
   failed-outbox accounting, and poison-row tests.
4. Define and test DST policy across planning, Calendar, reminders, recurrence,
   sessions, and digest/EOD scheduling.
5. Add a typed read-only explanation intent grounded in the last deterministic
   plan/outlook. Corrections must become explicit settings, task edits, or new
   proposals and must never become hidden behavioral memory.
6. Close accessibility and discoverability: captions/fallbacks, command menu,
   text-only alternatives, human date/time clarity, and a manual VoiceOver pass.
7. Choose and prove the supported local model baseline, expand the real-model
   corpus for recurrence/explanation/failure language, and document hardware
   expectations.
8. Run clean-install, upgrade, rollback, backup/restore, privacy, failure, soak,
   and seven-day daily-use rehearsals before the 1.0 release candidate.

## Deliberate exclusions for 1.0

- No Calendar writes or automatic task moves.
- No cloud task store, multi-user mode, or web dashboard.
- No behavior-derived productivity scoring, estimate learning, or hidden memory.
- No voice transcription until a local, explicit, resource-bounded privacy
  design exists; unsupported audio receives a clear text alternative.
- No broad autonomous messaging beyond the owner-approved digest, recap,
  reminders, adopted-session nudges, and once-version discovery.

## Final release-candidate gates

- No P0 or P1 finding above remains open, and every claimed fix has a named
  deterministic regression plus the appropriate manual/live evidence.
- Full Python suite and compile check pass from a clean environment; the exact
  count is recorded rather than treated as the quality claim.
- Complete real-model corpus passes on every declared supported model.
- Signed EventKit bridge builds and its grant/deny/revoke/privacy matrix passes.
- Ubuntu and macOS CI pass on the exact release commit and tag.
- Fresh install, v0.8/v0.9 upgrade, rollback, backup, restore, restart, sleep,
  DST, permanent queue failure, and unauthorized-access rehearsals pass.
- Seven-day privacy-safe daily-use evidence covers the complete loop and records
  qualitative usability findings and their disposition.
- Release version, package metadata, docs, handoff, schema, backup, tag, GitHub
  release, deployed process, and status all agree on 1.0.0.
- A final promise audit finds no claim that exceeds tested behavior, no known
  critical defect, and no misleading implication that planned, elapsed, or
  forecast work is complete or that Hob writes Calendar data.
