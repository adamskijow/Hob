# Hob 1.0 acceptance audit

This is the release-candidate evidence matrix for Hob 1.0. It is intentionally
started before v0.9 releases so missing proof cannot be reconstructed from
memory at the end. A green unit suite is evidence for the behavior a test names,
not proof of the broader customer journey.

Current baseline: v0.9.1 is released and healthy with 347 deterministic tests,
a 73-case real-model evaluation, signed EventKit build, and green Ubuntu/macOS
CI. The live v0.8 execution journey and v0.9.1 conversational-retraction replay
completed on 2026-07-11 with clean durable queues. Sustained v0.9 outlook and
EOD use remains. Mac App Store distribution is now an explicit parallel track;
its architecture decision and first native package exist, but no distributable
Store app is claimed yet.

## Release decision

**Not a 1.0 release candidate.** The product has strong local correctness and
privacy boundaries, but evidence is missing for sustained execution-loop use,
fresh-install timezone correctness, recoverable permanent queue failures, DST
edges, accessible end-to-end operation, supported-model quality, and a guided
durable install/upgrade journey. The LLM-native value also needs to progress
from natural intent classification to trustworthy explanation and negotiation
over deterministic plan/outlook results.

## Prioritized findings

### Resolved P0 parent gate: the v0.8 execution loop works live

The live database now contains the deliberately exercised v0.8 journey: initial
proposal, explicit adoption, status lookup, natural interruption, a separate
proposal while the old plan stayed active, explicit replacement, durable start
nudge, and direct reply anchored as `started`. Final aggregate state was one
active and one superseded run, three canceled old sessions, one started and two
planned revised sessions, with clean inbox/outbox queues. This closes the v0.8
parent gate. v0.9 still requires release and seven days of outlook/EOD daily use.

### P1: first-run time can be wrong by default

`HOB_TIMEZONE` defaults to UTC, while chat setup asks hours and days without
confirming timezone. A new macOS owner can finish setup successfully and receive
digests, plans, and reminders at the wrong local time. 1.0 must derive the
system timezone safely or make confirmation an unavoidable onboarding step,
with the result visible and changeable.

### P1: one poison delivery can stop the product indefinitely

Inbox and outbox processing is ordered and stops on the first failure, which is
correct for transient faults. A permanent parsing/store/Telegram failure is
never quarantined, so every later turn or reply can remain behind it forever.
Status shows pending work but offers no privacy-safe inspect/retry/quarantine
operation. Recovery must preserve at-most-once mutations and make any discard a
conscious local operator action.

### P1: installation is documented, not productized

`scripts/setup.sh` installs dependencies and runs doctor, but durable daily use
still requires copying and editing a plist, finding the Telegram user id,
loading launchd, configuring Hearth, and separately authorizing Calendar. A 1.0
install must validate the complete path, avoid secrets in files/logs, and offer
clear rollback/uninstall behavior.

### P1: DST behavior is unproven

Planning constructs timezone-aware wall times, but no tests cover nonexistent
spring-forward times or duplicated fall-back times. Reminders, fixed tasks,
Calendar periods, adopted sessions, and day boundaries need an explicit policy
and deterministic tests.

### P1: the supported-model promise is not evidence-backed

The runtime default and documentation offer `qwen2.5:7b-instruct`, while the
release eval gate runs only the 14B model. Either the supported baseline must
pass the complete corpus on target hardware or 1.0 must make 14B the declared
requirement and preflight resource expectations honestly.

### P1: LLM-native explanation stops one turn too early

Hob can understand “am I overloaded?” and compute the result safely, but it
does not have a typed way to answer “why did this not fit?” or negotiate which
assumption should change from the last proposal/outlook. 1.0 needs this grounded
loop to distinguish the agent from a scheduler without giving the model control
of time math or consent.

### P2: accessible fallback and release version drift were found early

This audit changed silent non-text Telegram updates into a clear text fallback
and lets media captions use the normal capture path. It also added a test tying
the package version to the runtime version. These fixes are prerequisites, not
proof of the full accessibility or upgrade journeys above.

### P2: sustained storage and logs have no operating envelope

Inbox, outbox, action, digest, and plan history are retained indefinitely and
launchd logs are unmanaged by Hob. This may be acceptable for one owner, but
1.0 needs a measured multi-year-size fixture, latency thresholds, documented
retention, and a safe log-rotation recommendation before making that promise.

## Coherent 1.0 scope

1. Finish, release, and dogfood v0.9 without weakening its explicit consent or
   read-only Calendar boundaries.
2. Make first install time-correct and durable: timezone confirmation, complete
   launchd install/update/rollback validation, Telegram credential check, and a
   useful doctor result before the owner expects a digest.
3. Add deterministic queue recovery with privacy-safe status, explicit local
   retry/quarantine commands, failed-outbox accounting, and poison-row tests.
4. Define and test DST policy across planning, Calendar, reminders, recurrence,
   sessions, and digest/EOD scheduling.
5. Add a typed read-only explanation intent grounded in the last deterministic
   plan/outlook. Corrections must become explicit settings, task edits, or new
   proposals and must never become hidden behavioral memory.
6. Close accessibility and discoverability: captions/fallbacks, command menu,
   text-only alternatives, human date/time clarity, and a manual VoiceOver pass.
7. Choose and prove the supported local model baseline, expand the real-model
   corpus for recurrence/explanation/failure language, and document hardware
   expectations.
8. Run clean-install, upgrade, rollback, backup/restore, privacy, failure, soak,
   and seven-day daily-use rehearsals before the 1.0 release candidate.
9. Advance the Mac App Store track without weakening the Open Local edition:
   create the Xcode-owned sandboxed bundle and consented helper, prove shared
   behavioral parity, complete no-Terminal onboarding and accessibility, and
   submit only after the Store-specific gates in `docs/app-store.md` pass.

## Deliberate exclusions for 1.0

- No Calendar writes or automatic task moves.
- No cloud task store, multi-user mode, or web dashboard.
- No behavior-derived productivity scoring, estimate learning, or hidden memory.
- No voice transcription until a local, explicit, resource-bounded privacy
  design exists; unsupported audio receives a clear text alternative.
- No broad autonomous messaging beyond the owner-approved digest, recap,
  reminders, adopted-session nudges, and once-version discovery.

## Final release-candidate gates

- No P0 or P1 finding above remains open, and every claimed fix has a named
  deterministic regression plus the appropriate manual/live evidence.
- Full Python suite and compile check pass from a clean environment; the exact
  count is recorded rather than treated as the quality claim.
- Complete real-model corpus passes on every declared supported model.
- Signed EventKit bridge builds and its grant/deny/revoke/privacy matrix passes.
- Ubuntu and macOS CI pass on the exact release commit and tag.
- Fresh install, v0.8/v0.9 upgrade, rollback, backup, restore, restart, sleep,
  DST, permanent queue failure, and unauthorized-access rehearsals pass.
- Seven-day privacy-safe daily-use evidence covers the complete loop and records
  qualitative usability findings and their disposition.
- Release version, package metadata, docs, handoff, schema, backup, tag, GitHub
  release, deployed process, and status all agree on 1.0.0.
- A final promise audit finds no claim that exceeds tested behavior, no known
  critical defect, and no misleading implication that planned, elapsed, or
  forecast work is complete or that Hob writes Calendar data.
