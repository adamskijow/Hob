# v0.9 product audit

Audit baseline: released v0.8.0 (`c369ebc`), schema 10, healthy in daily use.
The proposal/adoption loop completed its live customer journey on 2026-07-11,
including explicit replacement and a directly answered session nudge, so v0.9
can proceed to release and its own daily-use evaluation.

The v0.9 question is whether Hob can help before the day becomes impossible and
whether it can close an adopted day honestly. A list of this week's tasks is
ordinary scheduler behavior. The LLM-native value is understanding a natural
capacity question, computing the answer deterministically from private local
constraints, explaining the tradeoff, and accepting explicit corrections.

## Criteria

| Area | Evidence inspected | v0.8 finding |
| --- | --- | --- |
| User onboarding | `/setup`, first-plan activation, upgrade digest, adopt reply | Setup now reaches first adoption, but the first successful adoption does not teach the interruption/replacement boundary. An upgraded owner has not yet received the next-digest discovery note in real use. |
| Core usability | Week query, daily plan, adopted plan, EOD | “What is this week?” is a flat date-filtered list. It cannot answer whether the work fits. EOD ignores adopted sessions and asks about the entire on-deck list again. |
| Customer experience | Capture through deadline warning, morning execution, evening close | Deadline risk appears only after asking for a plan on the affected day. Hob cannot warn several days early, and an unfinished adopted session disappears into ordinary rollover without a specific evening review. |
| LLM-native differentiation | Natural constraints, planning, semantic recall | Hob understands natural capacity context and can explain one feasible day, but a conventional scheduler can match the remaining week view. Forward load, what-if constraints, and explicit negotiation are missing. |
| Bugs and failure states | Deadlines, dependencies, recurrence, Calendar denial, sleep | A deadline can be mathematically impossible across the remaining week without any signal. Dependencies must not be treated as complete merely because their effort was forecast. Unmaterialized future recurrence occurrences cannot be promised. |
| Privacy and safety | EventKit boundary, local model, persisted sessions | A weekly outlook can use seven opaque busy snapshots without event text. It must remain read-only, never create Calendar blocks, and never treat forecast allocation as consent or completion. |
| Operational reliability | Scheduler, outbox, backup, schema 10 | The existing schema is sufficient if outlooks remain ephemeral. EOD must retain once-per-day outbox deduplication and degrade to the existing recap if plan state is absent or corrupt. |
| Feature robustness | Fixed work, split work, default effort, buffers, dependencies, future dates | Multi-day allocation must reuse the same feasibility engine and assumptions as daily planning, account for remaining effort once, preserve fixed dates, and explain anything left over. |

## Prioritized findings

### P1: deadline and capacity risk is discovered too late

The feasibility core knows effort, deadlines, work hours, buffers, and Calendar
busy time, but only evaluates one requested day. Five hours due Friday can sit
quietly until Friday even when the remaining week has two free hours.

### P1: the adopted-day loop has no honest evening close

Session state records done, started, planned, and canceled blocks, yet EOD sends
the pre-v0.8 generic task list. It should distinguish completed sessions from
still-open planned work and ask what actually happened without marking anything
done from elapsed time.

### P1: week output does not answer the user's actual question

“Am I overloaded?”, “what will not fit?”, and “can I finish this by Friday?”
are classified near generic week or plan queries. A trustworthy answer requires
a typed outlook intent and deterministic horizon allocation, not an LLM guess.

### P1: working days are an unstated capacity assumption

The planning profile knows daily hours but applies them to every date. A weekly
outlook would therefore count Saturday and Sunday as ordinary capacity unless
the user manually constrained each request. Working days must be an explicit,
validated, inspectable, undoable onboarding choice before week math is credible.

### P2: first adoption lacks one crucial piece of onboarding

The adoption response explains that tasks and Calendar are unchanged and points
to plan status, but does not teach that drift creates a proposal and replacement
still needs explicit acceptance. That consent boundary should be explained once
at the moment it becomes relevant.

### P1 found during implementation audit: upgrades could lose weekend capacity

Adding a weekdays default to a profile that previously planned every day would
silently change an existing owner's behavior. Upgraded and older paired profiles
now retain and label the old all-days behavior until the owner explicitly
chooses, even when they predate onboarding-version metadata. Fresh setup records
even a skipped work-day answer so it cannot later look like a legacy profile.

### P1 found during implementation audit: natural what-if boundaries were leaky

The first implementation classified "by Friday" and weekly-budget questions but
still measured a rolling seven days and reapplied a weekly hour budget every day.
Named boundaries now shorten the deterministic horizon, while weekly budgets are
counted once across it. Literal morning and afternoon availability phrases shape
the actual time window rather than merely surviving as model context.

### P1 found during implementation audit: fixed work could appear movable

A timed item whose stated date had passed could become flexible on a later
forecast day. It now remains deferred with its missed time visible. Fixed-time
Calendar conflicts remain at their stated time and appear in the outlook's
conflict section.

## v0.9 scope

1. Add explicit working days to configuration, `/setup`, `/settings`, natural-
   language settings, undo, backup/export, and daily feasibility. Fixed work on
   a non-working day remains visible with a warning; flexible work does not fit.
2. Add a pure seven-day outlook that reuses daily feasibility, opaque Calendar
   snapshots, explicit work style, default estimates, and transition buffers.
   Allocate each task's remaining effort at most once across the horizon.
3. Preserve fixed do dates, waiting state, earliest starts, and dependencies.
   Forecasting a dependency does not mark it complete; dependent work may begin
   only on a later forecast day after the prerequisite fully fits.
4. Report daily used/open capacity, tasks left outside the horizon, deadlines
   that cannot fit, fixed-time conflicts, and every default-duration assumption.
5. Add a typed natural-language outlook query for overload, fit, deadline, and
   “what will not fit this week?” questions. The model may summarize only the
   deterministic result and cannot allocate or mutate work.
6. Make EOD plan-aware: show completed adopted sessions separately from still-
   open work, omit canceled sessions, and ask the user to state actual outcomes.
   Fall back to the existing recap when no adopted run exists.
7. Give the first successful adoption one short drift/replacement tip, recorded
   in inspectable local activation state and never repeated.

## Deliberate exclusions

- No automatic weekly schedule, plan adoption, task moves, or Calendar writes.
- No projection of future recurrence occurrences that do not yet exist as tasks.
- No inference that elapsed, forecasted, or planned work was completed.
- No behavior-derived estimates, productivity scores, streaks, or moralizing.
- No proactive extra weekly notification in v0.9; the outlook is user-requested
  until daily-use evidence supports an appropriate surface and cadence.

## Release gates

- Working-day defaults, custom weekdays, every-day profiles, invalid answers,
  onboarding resume/skip, undo, and fixed non-workday warnings are tested.
- Seven-day allocation, timezone boundaries, Calendar/break/buffer capacity,
  fixed work, split work, default effort, waiting, earliest starts, dependencies,
  and deadline risk have deterministic tests.
- A task's effort is never counted twice; forecast allocation never changes an
  item, proposal, adopted plan, reminder, Calendar, or action log.
- Dependency completion is simulated only for later forecast days and never
  persisted. Cycles and unresolved dependencies remain explicit risks.
- Natural overload, fit, and deadline questions pass real-model cases, while
  literal backstops correct safe model misclassification.
- Plan-aware EOD covers active, completed, partially completed, canceled,
  replaced, missing-task, restart, and no-plan fallback paths. It sends once.
- First-adoption coaching appears once and remains absent on replacement,
  cancellation, restart, and upgraded owners who already adopted.
- Backup/export/import, status, schema-9-to-10 migration, reminders, recurrence,
  undo/edit recovery, onboarding, and every v0.8 regression remain green.
- Full unit suite, real-model eval, signed Swift build, Ubuntu/macOS CI, backup,
  release, live restart, and post-deploy health verification pass.
- v0.8's real proposal/adoption/status/interruption/replacement loop is observed
  in daily use before v0.9 is tagged.

## Pre-release implementation evidence

- The pure forecast, typed outlook intent, five-step work-style onboarding,
  first-adoption coaching, and plan-aware EOD are implemented without a schema
  change. No v0.9 process has been deployed.
- The complete Python suite passes 343 tests. Coverage added for daily and
  horizon budgets, deadlines, fixed and split work, Calendar conflicts, missing
  session tasks, dependency cycles, timezone boundaries, working-day upgrades,
  setup skip/resume, EOD corruption fallback, and read-only state invariants.
- The signed native Calendar bridge builds successfully on macOS.
- PR #1 merged after exact release-head GitHub Actions run `29165242422` passed
  on Ubuntu and macOS; the macOS job also built and signed the EventKit bridge.
- The complete 72-case real-model evaluation passes on
  `qwen2.5:14b-instruct`, including work-day settings, two natural outlook
  shapes, and the full v0.8 language regression set.
- Implementation review corrected four misleading-success paths before release:
  silent weekend removal on upgrade, floating a missed timed commitment, using
  seven days for a named Friday boundary, and multiplying a weekly time budget
  by the number of days.
- A follow-up operations audit found that `status` could not prove historical
  adoption, replacement, completion, cancellation, or session-nudge delivery.
  It now exposes aggregate run/session states, latest adoption, and nudge counts
  without task labels, constraints, messages, Telegram ids, or secrets. Running
  live verification then proved the full v0.8 journey: one active and one
  superseded run, three canceled old sessions, one started and two planned
  revised sessions, durable nudge delivery/reply anchoring, and clean queues.
