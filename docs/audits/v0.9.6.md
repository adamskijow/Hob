# v0.9.6 numbered-exclusion and singleton audit

Date: 2026-07-15. Parent: v0.9.5.

## Customer outcome

`Finished it all except 1 and 6` now completes every other still-open item from
the exact displayed digest order. Rows that closed after the digest keep their
positions, and a missing or out-of-range position changes nothing instead of
sliding onto another task.

Only one local Hob process may own a Telegram bot, even when separate database
paths are selected. Unknown CLI arguments and ambiguous legacy/live database
selection fail before daemon startup.

## Root cause and production evidence

A local diagnostic invocation used the unrecognized `--help` argument. Hob had
no CLI rejection path, so it fell through to daemon startup. Configuration saw
the legacy checkout `hob.db` and selected it, while launchd correctly used the
Application Support database. The existing process lease was scoped to a
database path, so both processes could use the same Keychain Telegram token.

The legacy process sent and pinned Telegram message 223 at 13:56 with six stale
rows. The live database still recorded its correct four-row digest and message
219. The owner's message 225, `Finished it all except 1 and 6`, therefore
reached a context that could not contain position 6. The model proposed two
completions and a note. The earlier core backstop did not recognize bare `all`
or a numeric-only exclusion tail, so those proposals were applied.

The live action batch was backed up, reverted, and replaced atomically with the
intended result. Items 1 and 6 are open, items 2 through 5 are complete, the
spurious note is gone, and the repair is durable and undoable. Stale message 223
was unpinned and the real digest 219 restored.

## Deterministic evidence

1. The exact production mixed proposal completes only displayed items 2 and 4
   when 3 and 5 are already closed, leaving 1 and 6 open.
2. An exclusion outside the recorded digest applies no mutations.
3. Position 2 continues to mean digest row 2 after row 1 closes.
4. Named `everything but` completion repairs incomplete model proposals.
5. Two databases cannot acquire the same token-wide Telegram lease.
6. Distinct Telegram tokens remain independently runnable.
7. Lock paths contain no raw token and use mode `0600` inside a private
   directory.
8. `--help` and unknown commands never load daemon configuration.
9. Daemon and status paths refuse ambiguous database selection.

## Remaining risks

1. A remote copy of Hob on another Mac cannot share the local file lease.
   Telegram will still reject competing `getUpdates`, but multi-host ownership
   is intentionally unsupported and should be documented before 1.0.
2. A message referring to a genuinely different list without saying `that
   list` can still require clarification. Hob prefers no change over shifting a
   displayed number.
3. The Store edition must apply the same exact displayed-list identity when its
   conversational transport is enabled.

## Decision

Release as v0.9.6 after Python, lock, syntax, real-model, Swift, signed EventKit,
Xcode, exact-commit Ubuntu/macOS CI, backup, deployment, and live singleton
verification pass.
