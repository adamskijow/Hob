# v0.9.4 coordinated-completion audit

Date: 2026-07-14. Parent: v0.9.3.

## Customer outcome

A natural report such as “I did home insurance and hit the grift” now marks
both named tasks complete. Hob no longer turns the second task into “Next” just
because its verb has the same present and past form. If the owner actually
changes intent, as in “I did home insurance and will hit the grift,” “started,”
or “worked on,” the unfinished task remains open.

## Root cause and production evidence

The real-model interpreter returned one `complete` action and one `start`
action. The deterministic core accepted both because each referenced a valid
open item with high confidence. The live action log therefore contains only
the home-insurance completion for the inbound screenshot message, while the
second named task remains open. This was a tense-scoping error, not a Telegram
delivery, storage, or reference-resolution failure.

## Increment audit

| Criterion | Evidence | Assessment |
| --- | --- | --- |
| User onboarding | The once-per-version digest note demonstrates a two-task completion report and states that future or partial-progress wording stays open | Existing owners discover the correction in the normal loop; fresh setup gains no extra step. |
| Usability | One ordinary sentence closes both tasks; no separate “done” follow-up is required | Matches how people naturally answer the EOD prompt. |
| Customer experience | The reply lists both tasks as done and no longer contradicts the owner with a false “I have not marked it done” message | Repairs a trust-breaking acknowledgement mismatch. |
| Bugs and robustness | A deterministic reconciliation guard corrects `complete + start` only for an explicit opening completion tense with coordinated clauses and no scope-breaking wording | Robust to model drift while conservative about unfinished work, without prompt tuning that could shift unrelated intents. |
| Privacy and safety | The correction uses the inbound sentence and already-proposed task ids only. It stores no new context and makes no additional model call | No broader data exposure or hidden memory. |
| Delivery reliability | Both completions use the existing atomic mutation batch, durable inbox, action log, outbox, and undo path | A crash cannot leave the corrected turn half-applied. |
| Accessibility | The behavior is entirely text-driven and the reply explicitly names each completed task | No reaction, button, pin, color, or visual-only affordance is required. |
| LLM-native behavior | The model still performs semantic task matching and action proposal; deterministic grammar and intent boundaries constrain a known tense ambiguity | More capable than exact-command scheduling without making the model the final authority. |

## Deterministic evidence

1. The exact production wording converts a model-proposed completion plus start
   into two completion mutations and no start.
2. The complete MessageService path marks two stored tasks done and replies
   with both names.
3. “I did A and will do B” preserves the second start.
4. “I did A and worked on B” preserves partial progress rather than claiming
   completion.
5. The real `qwen2.5:14b-instruct` corpus includes the screenshot wording and
   resolves both expected task ids.

## Remaining risks

1. Coordinated language is open-ended. Uncovered contrast or aspect wording may
   still need a clarification instead of a silent correction.
2. The deterministic guard intentionally repairs only a model-produced
   `complete + start` split. Other malformed action combinations fail through
   existing reference and confidence guards rather than broad text parsing.
3. The Mac App Store runtime does not yet connect its native model
   interpretation pipeline, so this guard must become a shared parity case when
   that seam is implemented.

## Decision

Release as v0.9.4 after the full Python, Swift, real-model, EventKit, Xcode,
lock, syntax, and exact-commit CI gates pass. Back up the live database before
restart and verify version, schema, queues, owner pairing, and model health.
