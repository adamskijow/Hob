# v0.7 product audit

Audit baseline: released v0.6.0 (`bfdfee7`), schema 9, running in daily use.

This audit is a release gate, not a feature inventory. Each finding is evaluated
from the first-run customer journey through sustained use, recovery, and upgrade.

## Criteria

| Area | Evidence inspected | v0.6 finding |
| --- | --- | --- |
| User onboarding | `/start`, setup script, README, Calendar authorization, `/settings` | Pairing succeeds, but the product does not guide a new owner through a usable planning profile, show progress, resume setup, or verify readiness in chat. Critical preferences are discoverable only in documentation. |
| Core usability | Capture, digest, plan output, follow-up context, commands | Planning produces a realistic timeline, but its numbered order is not the canonical task-list order. A follow-up such as “do the second one” can therefore affect a different task. |
| Customer experience | Capture to plan to correction loop | Unknown durations are visibly estimated, but the hard-coded 30-minute assumption cannot be personalized. Plans also place work directly against adjacent commitments with no user-controlled transition time. |
| LLM-native differentiation | Interpreter, planning explanation, conversational correction | Natural-language planning is strong, but Hob does not yet remember explicit work-style instructions beyond hours and one break. A generic scheduler can match the remaining experience. |
| Bugs and failure states | Focus expiry, restart persistence, malformed model output, Calendar denial | Calendar denial degrades safely. Setup has no resumable state because no guided setup exists. Plan ordinals are a correctness bug, not merely missing polish. |
| Privacy and safety | EventKit bridge, confirmation path, local persistence | Calendar titles and identifiers stay outside Python and the model. Preference memory must remain explicit, inspectable, local, undoable, and never inferred silently from behavior. |
| Operational reliability | Status, backup, launchd, schema, CI | v0.6 is healthy. v0.7 should avoid a schema migration by using transactional meta plus the existing action log for settings. |
| Feature robustness | Durations, buffers, plan diffs, dependencies, splitting | The feasibility core has deterministic gates, but default effort and transition space are not inputs, and plan follow-up references lack a deterministic contract. |

## Prioritized findings

### P1: first-run setup does not reach a known-good customer state

A new owner receives a broad welcome but no short, resumable sequence that
establishes working hours, protected time, estimate defaults, or transition
space. There is no in-chat setup status. This makes the first useful plan depend
on reading external documentation and remembering special phrases.

### P1: plan ordinals can target the wrong task

The plan numbers chronological blocks, while deterministic reference resolution
uses the canonical digest/list order. Conversational focus stores mutations but
not a generated plan. The visible number and the applied target can diverge.

### P1: explicit work-style memory is too shallow

Working hours and one break are remembered, but the two most consequential
assumptions remain global constants: unknown effort is 30 minutes and no space
is left between work or Calendar commitments.

### P2: Calendar activation is terminal-only

Apple permission must remain an explicit local-user action. The in-chat setup
should report the state and give the exact command, but must not imply that a
Telegram reply can grant macOS private-data access.

### P2: a proposed plan cannot yet be adopted as structured sessions

Persisting one due time cannot faithfully represent split work. v0.7 must not
offer a partial “lock in this plan” feature that silently discards segments.

## v0.7 scope

1. Add a short `/setup` journey that is safe to skip, persists and resumes after
   restart, and reports completion in `/settings`.
2. Guide explicit, undoable choices for working hours, protected break, default
   estimated duration, and transition buffer.
3. Feed default duration and transition buffer into deterministic feasibility;
   expose every assumption in plan output and settings.
4. Persist plan order as conversational focus and deterministically resolve
   numbered follow-ups against what the user just saw.
5. Report Calendar readiness during setup without requesting permission from
   Telegram or weakening the working-hours fallback.

## Deliberate exclusions

- No implicit preference learning from behavior.
- No Calendar writes.
- No plan-adoption command until split sessions have a first-class data model.
- No broad free-form memory that cannot be inspected or deterministically used.

## Release gates

- Fresh `/start`, explicit `/setup`, skip, cancel, resume-after-restart, and
  completion paths are covered end to end.
- Every stored preference is visible, validated, undoable, exportable through
  existing meta export, and has a deterministic fallback.
- Unknown-duration plans use the configured default and label it as an estimate.
- Transition buffers prevent adjacent task/task and task/Calendar placement
  without moving fixed commitments.
- Plan ordinal follow-ups resolve to the displayed plan order, including after a
  restart within the focus window.
- Existing capture, correction, recurrence, reminders, recovery, privacy, and
  v0.6 feasibility tests remain green.
- Real-model cases cover every new natural-language setting and plan follow-up.
- Linux and macOS CI, native Swift build, backup, release, live restart, and
  post-deploy health verification pass.

## Post-release evidence

- Local gates passed on 2026-07-11: 304 unit/integration tests, Python compile
  verification, signed Swift bridge build, and 63/63 real-model cases on
  `qwen2.5:14b-instruct`.
- [GitHub Actions run 29153235030](https://github.com/adamskijow/Hob/actions/runs/29153235030)
  passed on Ubuntu and macOS; macOS also built the EventKit bridge.
- A consistent pre-release backup was written and verified before restart at
  `backups/hob-pre-v0.7.0.db` in Hob's Application Support directory.
- The release backup rehearsal exposed two coexisting database paths. Recovery
  commands now refuse that ambiguous state unless `HOB_DB_PATH` explicitly
  selects one, report their source path, and ignore migration backups in Git.
- Live restart preserved schema 9 and five open tasks. Inbound, outbound, and
  failed queues were empty; the paired owner, digest state, and model were
  healthy; launchd reported a running process with a clean start.
- Calendar remained `not_determined`, and status confirmed the explicit
  working-hours-only fallback. No remote permission request was attempted.
- Sustained daily-use evidence begins with this deployment. Setup progress,
  plan-follow-up accuracy, and buffer usefulness remain explicit observations
  for the next increment audit rather than assumed success.
