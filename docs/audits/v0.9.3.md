# v0.9.3 Telegram service-event audit

Date: 2026-07-14. Parent: v0.9.2 plus merged Store-runtime work through PR #14.

## Customer outcome

Hob no longer sends an unsupported-media reply when Telegram reports that Hob
pinned the morning digest. Pin, membership, chat-setting, and other Telegram
status events still enter the durable inbox and advance the polling offset, but
they are completed as silent no-ops and never enter the owner-message handler.
An actual uncaptioned owner attachment still receives clear text guidance.

## Root cause

The Telegram adapter previously treated every `Message` without `text` or a
media `caption` as unsupported owner content. Telegram represents pinning as a
service `Message` with `pinned_message` and no text, so the adapter could not
distinguish Hob's own pin side effect from an uncaptioned photo, voice note, or
other real owner input.

The screenshot evidence is consistent with that sequence: the 07:00 digest was
sent and pinned, followed immediately by the unsupported-message reply without
an owner message between them.

## Deterministic evidence

1. A lightweight digest-pin update calls no handler, sends no reply or typing
   action, and advances the durable offset.
2. A real `telegram.Update` containing a real pinned-message status object takes
   the same silent path through the library classifier.
3. A separate membership status update is silent, guarding the general class
   rather than only the observed pin field.
4. An uncaptioned attachment continues to receive the documented fallback and
   advance the offset.
5. Captioned media continues through the normal owner-message path.

## Remaining risks

1. Telegram can add future service types. The pinned library version's aggregate
   filter will cover types it knows; dependency upgrades must keep the fallback
   list and regression aligned.
2. This does not make Telegram reply presentation exactly once. A process death
   after Telegram accepts a reply but before Hob marks it sent can still repeat
   the reply, as documented in the Store delivery-pipeline audit.
3. The screenshot proves the symptom and timing, not the raw update JSON. The
   deterministic real-object test proves the matching Bot API representation.

## Decision

Release as v0.9.3, back up the live database, restart the launchd service, and
verify healthy queues. The next morning digest is the natural end-to-end proof
that pinning remains silent in daily use.
