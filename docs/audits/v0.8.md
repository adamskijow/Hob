# v0.8 product audit

Audit baseline: released v0.7.0 (`f715eed`), schema 9, running in daily use.

The v0.8 question is not whether Hob can draw a plausible schedule. It is
whether a person can deliberately adopt that schedule, understand what is
current, recover from interruption, and trust that conversational changes do
not silently mutate tasks or Calendar.

## Criteria

| Area | Evidence inspected | v0.7 finding |
| --- | --- | --- |
| User onboarding | Fresh and returning `/start`, `/setup`, digest, `/settings`, first plan | Fresh setup is guided and resumable, but an upgraded owner receives no in-product discovery in the normal digest. Setup completion asks for a first task and plan without recording whether that success happened. |
| Core usability | Plan proposal, focus expiry, “start”, reminders, restart | A plan remains chat text plus 15-minute focus. There is no durable answer to “what am I doing now?”, no explicit adopt/cancel state, and no safe distinction between a proposal and the day being followed. |
| Customer experience | Morning digest through execution and EOD | The user must mentally carry the proposed timeline. Completion updates the task list but not a visible plan state; interruption requires generating another proposal with no clear replacement step. |
| LLM-native differentiation | Natural constraints, plan explanation, correction loop | Hob understands constraints better than a fixed scheduler, but stops before the valuable loop: natural deviation, deterministic feasibility repair, an explained diff, and explicit acceptance. |
| Bugs and failure states | Date intent, split work, stale plans, duplicate delivery | “Plan tomorrow” is classified as planning but the requested date is ignored; the feasibility engine always targets today. Persisted proposals can contain completed tasks until another plan is generated. |
| Privacy and safety | Local model, EventKit boundary, plan persistence | Adopted sessions can stay local and must never become Calendar writes. Adoption and replacement must be explicit; Hob must not infer consent from viewing, starting, or completing one task. |
| Operational reliability | Schema migration, backup/export/import, outbox, launchd | First-class sessions require a migration and portable export support. Start nudges must use the durable deduplicated outbox and survive restart without double delivery. |
| Feature robustness | Split sessions, timezone, recurrence, completion, replan | A task may have several plan blocks, so task due fields cannot represent adoption. Session identity, status, supersession, and task lifecycle behavior need deterministic contracts. |

## Prioritized findings

### P1: proposals cannot become an executable day

The visible plan has no durable identity or state. “Use this plan” cannot be
represented faithfully, especially for split work, and “what is next?” may
generate a different proposal instead of reporting the plan already accepted.

### P1: planning a named future day silently plans today

The query schema carries a date intent, but plan reconciliation returns before
resolving it and the feasibility engine derives its target from `now`. A result
that confidently labels tomorrow's request with today's availability is a trust
failure.

### P1: deviation has no acceptance boundary

“I lost an hour” can create a revised proposal, but there is no explicit replace
operation. Hob must show the diff and wait for adoption rather than silently
changing the active day or leaving the user unsure which version is current.

### P2: upgraded owners can miss product onboarding

Returning owners are correctly not forced through setup, but daily use does not
surface a concise one-time path to the new profile and execution workflow.
Lifecycle onboarding should use the existing digest, appear once per release,
and remain dismissible.

## v0.8 scope

1. Resolve and validate the requested planning day before feasibility; support
   today, tomorrow, and explicit named dates without using future Calendar event
   text or inventing capacity.
2. Add first-class local plan runs and sessions. Preserve every split block,
   its source task, time, segment, state, and the proposal it came from.
3. Add explicit natural-language adoption, replacement, and cancellation.
   Adopting sessions never changes task dates, creates Calendar events, or marks
   work complete.
4. Make “current plan”, “what is next?”, and “start the second one” read the
   adopted run deterministically. Task completion or dropping closes remaining
   sessions for that task without changing recurrence semantics.
5. Deliver at most one durable start nudge per adopted session. A reply remains
   anchored to the underlying task; restart and retry cannot duplicate it.
6. Replanning while a run is active produces an explained proposal diff. The
   active run changes only after an explicit replace response.
7. Add one concise, once-per-version digest note for an upgraded owner, linking
   setup status to the first adoptable plan. Fresh onboarding remains the normal
   path and no feature tour interrupts ordinary task capture.

## Deliberate exclusions

- No Calendar writes or automatic meeting creation.
- No silent adoption, behavior-derived preferences, or inferred completion.
- No multi-user collaboration or shared plan state.
- No week-level auto-scheduling; future-day correctness lands before broader
  horizon planning.
- No motivational scoring, streaks, or notification volume beyond explicitly
  adopted session starts.

## Release gates

- Today, tomorrow, explicit-date, past-date, and timezone-boundary planning have
  deterministic tests; a named day can never silently fall back to today.
- Proposal, adopt, replace, cancel, current, next, split-session, restart, and
  stale-task paths are covered end to end.
- Adoption is atomic and idempotent. A replayed Telegram update cannot create a
  second run, and replacing a run cannot leave two active versions.
- Completion, drop, recurrence rollover, undo, and edited-message behavior do
  not orphan active sessions or falsely reopen them.
- Session nudges use the durable outbox, deliver once, retain reply anchoring,
  and degrade quietly when the model or Telegram is unavailable.
- Backup, export/import, released-schema migration, status, and restore preserve
  or safely report plan-session state.
- Upgraded-owner discovery is one-time, dismissible, and tested separately from
  fresh onboarding and ordinary digest delivery.
- Existing privacy promises remain true: EventKit supplies opaque busy periods,
  the model sees no Calendar text, and adoption performs no external writes.
- Full unit suite, real-model eval, signed Swift build, Ubuntu/macOS CI, backup,
  release, live restart, and daily-use verification pass.

## Daily-use evidence to collect

- Whether the v0.7 setup prompt is discoverable and the four answers feel short.
- Whether default effort and transition space improve the first real plan.
- Whether visible plan ordinals continue to match conversational follow-ups.
- Whether an adopted plan reduces “what should I do?” friction without adding
  noisy nudges or making the user feel controlled by stale time blocks.

  integrity-checked `backups/hob-pre-v0.8.0.db`: schema 9 with five open tasks.
  Migration also created its automatic schema-9 safety copy beside the database.
- Live restart migrated to schema 10 and preserved all five open tasks. Inbound,
  outbound, and failed queues were empty; the paired owner, digest state, and
  model were healthy; launchd reported a running process and a clean v0.8 start.

  remained `not_determined`; no permission request or external write occurred.

  status check, one natural interruption/replacement, and observation of start-
  nudge usefulness. The upgraded-owner digest note is deliberately pending the
  next normal digest rather than being sent as an extra release notification.
