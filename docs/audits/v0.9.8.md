# v0.9.8 model-native recap audit

Date: 2026-07-19. Parent: v0.9.7.

## Customer outcome

An answer to Hob's evening recap is conversational again. The owner can say
`nada`, `today was a wash`, use an unseen metaphor, report partial progress, or
name completed work. Hob interprets the meaning in the active recap context
instead of requiring one of a few accepted phrases. A true zero-result answer
changes no tasks and says that the displayed work remains open.

The prompt, `/help`, feature guide, and once-per-version discovery note now say
to answer naturally and explicitly state that no fixed phrase is required.

## Why v0.9.7 was not good enough

v0.9.7 fixed the production failure for `Nothing got done` with a deterministic
regular-expression allowlist before the model call. That made the exact report
reliable during an outage, but recreated the command-parser failure Hob exists
to avoid. A real-model probe without literal prompt examples confirmed the
problem: the 14B interpreter understood `today was a wash` but classified
`nada` as chitchat.

The issue was architectural, not a missing synonym. Adding more strings would
have expanded an unbounded language problem one phrase at a time and kept the
LLM out of the product moment where it adds value.

## Contract and safety boundary

1. The main local model receives the machine-owned kind of the most recently
   presented list and may propose a typed `Recap(outcome="none")` action.
2. When a current evening recap exists and the main result is recap, chitchat,
   or unknown, a narrow second model pass classifies the conversational outcome
   as `none`, `social`, or `other`. It reasons about meaning and contains no
   runtime phrase table.
3. The second pass is required even for a direct recap proposal. The full eval
   caught the main model temporarily treating gratitude as a recap answer; the
   focused pass now distinguishes a work claim from a merely social message.
4. Concrete actions such as capture, completion, note, query, or reschedule are
   never replaced by this pass. If a model emits recap beside a concrete action,
   the planner discards recap and reconciles the concrete action.
5. The deterministic planner accepts a zero outcome only with a nonempty,
   unexpired list whose provenance is exactly `eod`, with no forwarded sender or
   pending onboarding/clarification answer. It then returns an acknowledgment
   with no mutation batch.
6. If either required model call fails, Hob reports the model outage and changes
   nothing. Availability is not faked with a brittle language fallback.

This restores the intended division: the model owns semantic understanding;
the core owns provenance, confidence, allowed effects, and atomic state change.

## Verification evidence

- 407 deterministic Python tests cover parsing, context propagation, direct and
  recovered recap outcomes, mixed actions, social speech, concrete tasks,
  partial progress, setup/clarification precedence, forwarding, confidence,
  initial and between-pass outages, mutation-free acknowledgment, and undo.
- The 84-case `qwen2.5:14b-instruct` corpus includes literal, multilingual,
  idiomatic, unseen metaphorical, social, partial-progress, concrete-action,
  and wrong-provenance cases.
- The full native, platform, exact-commit CI, backup, deployment, and live
  verification gates remain required before the release is considered shipped.

## Remaining risks

1. Ambiguous or zero-result replies after an evening recap can require two local
   generations, increasing latency. The scope is narrow and correctness is more
   important than pretending an uncertain first classification is final.
2. A model outage means the recap answer must be retried. This is an honest
   consequence of model-owned semantics and is preferable to a hidden phrase
   parser that works only for known wording.
3. Presented recap context expires after 24 hours and is filtered to still-open
   ids. A late or fully closed recap answer will ask for clarity rather than
   claim success.
4. The Store edition still needs this contract implemented and evaluated with
   its on-device model before feature parity can be claimed.

## Decision

Release as v0.9.8 only after the complete Python suite, 84-case 14B evaluation,
lock and syntax checks, Swift tests, signed EventKit build, Xcode Store-shell
build, exact-commit Ubuntu/macOS CI, live-database backup, deployment, and live
health verification all pass.
