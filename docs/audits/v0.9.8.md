# v0.9.8 model-owned language audit

Date: 2026-07-21. Parent: v0.9.7.

## Customer outcome

Hob no longer has a second, hidden command language inside its conversational
interface. The owner can answer an evening recap with `nada`, tell a stale task
`it needs to stay on`, approve or revise a confirmation, keep an onboarding
default, say `I'm wiped`, remove lunch, or retract a turn in ordinary language.
The local model owns those meanings. The deterministic core confines any effect
to known task ids, the current machine-owned question, a validated typed value,
or the exact displayed set.

This closes the production bug where `It needs to stay on` was not tied to the
active stale-task question and was misread as changing the evening recap to
20:00. It also removes the broader architectural cause instead of adding that
sentence to an allowlist.

## Whole-program finding

The recap fix exposed the same anti-pattern elsewhere:

- confirmations used a `startsWith("yes")` shortcut, so unrelated text such as
  `yesterday...` could approve a held mutation;
- stale-task and onboarding choices used exact phrase tables before the model;
- planning constraints, retractions, break removal, bulk exclusions,
  coordinated completion, recurrence recovery, waiting conversion, and plan
  actions had English-specific repair rules after the model;
- raw planning keywords could turn captures such as `add capacity review to my
  list` into queries;
- date and deadline backstops could replace typed model intent by rescanning the
  sentence.

Those paths are removed. Explicit slash commands, Telegram callback payloads,
reactions, item ids, and displayed ordinal positions remain deterministic
because they are closed protocols rather than free-form language.

## Contract and safety boundary

1. Every free-text turn reaches the primary local model.
2. The model returns typed settings, dates, planning constraints, bulk scope,
   recap outcomes, nudge decisions, confirmation decisions, onboarding
   decisions, and task actions.
3. Narrow semantic passes audit machine-owned recap/nudge/confirmation/setup
   context, capture/plan/undo routing, capture details, settings, existing-task
   schedule metadata, and bulk scope/exclusions. A compact independent vote
   resolves only route disagreements. No pass can invent an existing-task id.
4. Core reconciliation verifies prompt freshness and kind, open ids, displayed
   membership, allowed outcomes, confidence, literal setting-value grounding,
   value bounds, date arithmetic, destructive holds, and atomicity.
5. A contextual decision cannot ride beside a concrete action. A revised or
   conditional instruction is handled as the new instruction, not approval.
6. A model error is not widened by a phrase repair. Invalid proposals ask,
   confirm, or change nothing.
7. The model classifies date meaning; the core performs calendar arithmetic.
   The core no longer invents or replaces typed date/deadline intent from raw
   prose.

## Verification evidence

- 417 deterministic Python tests cover interpreter parsing, focused semantic
  adjudication, provenance, typed constraints, failure behavior, application,
  undo, durable state, Telegram edges, and scheduling.
- The real-model corpus expands from 84 to 102 cases, adding stale-task
  paraphrases, confirmation collisions and revisions, onboarding language,
  planning idioms, capture/query collisions, break removal, and retractions.
- The unused raw-text date/deadline backstops and their `dateparser` dependency
  are removed, preventing dormant phrase parsing from returning to the mutation
  path.
- Setting undo now restores an absent metadata key as absent rather than an
  empty override, preserving the configured default exactly. The live database
  was integrity-checked, backed up, and verified batch `b50` was restored from
  `"20:00"` to its recorded null pre-state.
- Full Python, 102-case 14B model, lock/syntax, Swift, signed EventKit, Xcode,
  exact-commit hosted CI, backup, deployment, and live-health gates are required
  before the release is considered shipped.

## Remaining risks

1. Contextual questions and bulk operations can require two local generations,
   increasing latency. Correct meaning is more important than hiding uncertainty.
2. Supported-model quality must be proven for every documented model. The
   release gate currently exercises the 14B model, while 1.0 still needs an
   honest minimum-model decision.
3. The Store edition needs the same typed contextual contracts and Apple-model
   corpus before parity can be claimed.
4. Hob still needs the separate 1.0 explanation/negotiation loop, timezone and
   durable install work, queue recovery, DST evidence, and sustained daily-use
   proof tracked in the 1.0 acceptance audit.

## Decision

Release as v0.9.8 only after every gate above passes, the live database is
backed up, the accidental `eod_time` mutation is restored from its recorded
pre-change value, the daemon runs the exact release commit, and the production
phrase is verified without changing a setting.
