# v0.9.7 zero-completion recap audit

Date: 2026-07-18. Parent: v0.9.6.

## Customer outcome

An evening answer such as `Nothing got done` is now a valid report. Hob changes
no tasks and replies that the displayed items remain open on deck. The same
path accepts common equivalents including `none`, `no progress`, `I didn't get
anything done`, and `none of them were completed`.

This outcome is recognized before the local model is called. It therefore
remains available during a model outage and cannot be converted into a capture,
completion, note, or other mutation by an incorrect model proposal.

## Production evidence and root cause

At 21:35 on 2026-07-18, the owner answered the two-item evening recap with
`Nothing got done`. The released interpreter returned an unknown action, and
Hob replied `I did not catch a task there. Can you rephrase?`

The durable inbox records the exact owner message as completed, the outbox
records the incorrect clarification, and the action log has no corresponding
batch. Both displayed tasks remained open. The failure was therefore a missing
zero-result conversational outcome, not state corruption.

## Deterministic evidence

1. The exact production phrase returns a two-item acknowledgement and changes
   no task.
2. The message is handled without calling the LLM.
3. Strong variants override even a model-proposed completion.
4. `none` works with a recent proactive list but remains unknown without that
   context.
5. A mixed negative and positive report is not swallowed by the zero-result
   detector.
6. No action batch is created, so `/undo` continues to refer to the previous
   real change.
7. The proactive recap teaches the supported response in both task-list and
   adopted-plan forms.

## Remaining risks

1. Keeping work open does not automatically reschedule it. The next digest and
   planning request still decide when it belongs.
2. Partial-progress reports remain a separate interpretation path. They should
   preserve useful notes without implying completion and need continued daily
   use before 1.0.
3. A terse answer after the proactive-list context expires asks for clarity.
   This deliberately prefers no assumption over an unanchored `none`.
4. The Mac App Store runtime does not yet expose the full conversational recap
   loop, so parity remains a Store release gate.

## Decision

Release as v0.9.7 after the full Python suite, lock and syntax checks,
real-model evaluation, Swift tests, signed EventKit build, Xcode Store-shell
build, exact-commit Ubuntu/macOS CI, backup, deployment, and live health
verification pass.
